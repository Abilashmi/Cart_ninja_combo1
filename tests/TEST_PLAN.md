### What this does NOT cover, and why (permanent, not a to-do list)

Everything below requires a live Shopify embedded-admin session. Three independent approaches were tried and all hit the same wall — this isn't a skill/tooling gap, it's how Shopify's embedded auth is designed:

1. **`storageState` replay** — fails because embedded apps authenticate via short-lived (~60s) App Bridge JWT session tokens generated by live JS in a real Shopify iframe, not a static cookie that can be snapshotted and replayed later.
2. **CDP-attach to an already-logged-in real Chrome** — got real visual access (screenshots) to the authenticated admin page, but Playwright's CDP frame-tracking cannot see into the cross-origin embedded iframe's DOM (a Chrome site-isolation limitation, confirmed even with `--disable-features=site-per-process`). No interaction possible.
3. **Session cookie extraction** — there is no session cookie at all for the app's own domain; auth is 100% token-based and JS-driven.

**Blocked as a result:**
- The entire admin UI (Dashboard, Cart Drawer editor's Polaris screens, Coupon Banner editor, FBT editor, Setup Guide, Account, Subscribe/Pricing) — can't click through any of it.
- Analytics, Billing, Bundles/Combo Forge, Discounts — confirmed their `api.*.jsx` Node routes all call `authenticate.admin()`, same wall as the UI. There is no PHP-direct bypass for these the way there is for Cart Drawer/Progress Bar/Coupon Slider/Upsell/FBT/AI (those have a parallel PHP-only path that Node happens to also proxy through).
- BrixBar live chat completion itself (`api.ai.chat.jsx`, `api.ai-agent.*.jsx` turn routes) — also behind `authenticate.admin()`. Only the chat *history* persistence (`ai_conversations.php`/`ai_messages.php`) and the *action-execution* engine (`ai_agent_apply.php`) are PHP-direct and covered.

**If you want coverage here anyway**, the only paths forward are: (a) a dedicated test-mode auth bypass added to the app itself (requires your explicit approval to touch `app/` code), or (b) Shopify's own testing infrastructure if any exists for embedded apps (not something I have visibility into beyond what's documented publicly).

### Known real findings from building this suite (not hypothetical)
1. `X-Forge-Secret` check is inert on local XAMPP — `config.php` never sets `SHOPIFY_API_KEY` in PHP's process env, so `if ($expected && ...)` always skips. Works correctly on the deployed remote.
2. `coupon_slider_settings.php` silently NULLs `selected_coupons` on any POST that omits the field (`ON DUPLICATE KEY UPDATE ... = VALUES(...)` uses whatever this request sent, not the previous value).
3. Storefront tests must run single-worker (`storefront` project, `workers: 1`) — concurrent workers against the live production store caused flaky timing on the drawer's config fetch.
4. `window.__CC_DRAWER_API` (referenced in `CLAUDE.md` as how `Fbt.liquid` opens the drawer) does not exist in the current `cart_drawer_inline.js` — stale documentation, flagged separately.
5. `app/services/ai-agent-actions.server.js` and `ai-agent-history.server.js`, documented in `CLAUDE.md` as the AI action/history layer, **don't exist** — that logic lives in PHP (`ai_agent_apply.php`, `ai_conversations.php`, `ai_messages.php`) called via thin Node proxies.
6. `fbt_widget.ai_product_count` is `NOT NULL` with no default — any POST to `save_fbt_widget.php` omitting `aiProductCount` fails with a DB integrity error, not a friendly validation message.

## 1–5. Original discovery & plan (historical record)

### 1. Inventory summary

Full detail came from three parallel codebase surveys. Condensed findings:

#### 1.1 Admin pages (behind Shopify auth, `app/routes/app.*.jsx`)
Dashboard (`app._index.jsx`), Cart Drawer editor (`app.cartdrawer.jsx` — the core feature), Setup Guide (`app.setup.jsx`), two separate discount-list UIs (`app.discount.jsx` and `app.coupons.jsx` — likely one is legacy, needs your confirmation), Combo/Bundle dashboard + builder (`app.bundles._index.jsx`, `app.bundles.customize.jsx` — 5500+ lines, `app.bundles.analytics.jsx`, `app.bundles.templates.jsx`), Billing (`app.billing.jsx`), Account (`app.additional.jsx`), Analytics (`app.analytics.jsx`), Pricing/Subscribe (`app.subscribe.jsx`), Coupon Banner editor (`app.productwidget.jsx`), FBT editor (`app.fbt.jsx`), Brix AI page (`app.brix-ai.jsx`).

**Dead/suspicious code flagged by the survey** (excluded from the plan unless you say otherwise):
- `app.cartdrawer.old.jsx` — looks like a superseded duplicate of the real cart drawer editor.
- `app.upsell.jsx` — no nav entry, component renders `null`.
- `app/routes/store.jsx` — static mock page, not wired to the real app.
- Four orphaned JS assets in the theme extension (`cart_drawer.js`, `cartdrawer.js`, `fbt_widget.js`, `coupon_slider.js`) not referenced by any `asset_url` — only `cart_drawer_inline.js` and the inline `<script>` in `coupon_slider.liquid`/`Fbt.liquid` are actually live.

#### 1.2 API routes (`app/routes/api.*.jsx`)
~40 endpoints. Two pairs of near-duplicate names worth flagging: `api.cartdrawer-config.jsx` vs `api.cart-drawer-config.jsx` (different tables — legacy `cart_drawer` blob vs normalized `cart_drawer_config`). All write to either MySQL (`cart_drawer_ninja`) directly or proxy to PHP. **Confirmed: every one of these calls `authenticate.admin()`.**

#### 1.3 Webhooks
Order lifecycle (`orders.create/updated/cancelled/paid`, `refunds.create`), cart activity, GDPR compliance (with a likely-redundant pair: `webhooks.compliance.jsx` vs the three individual `webhooks.customers.*`/`webhooks.shop.redact.jsx` files), subscription/billing updates, app uninstall.

#### 1.4 PHP backend (`php_backend/*.php`)
~30 files. Six are called by Node (`api-helpers.js`) but **only exist on the remote deployment** (`shop.php`, `discount.php`, `templates.php`, `visitors.php`, `clicks.php`, `orders.php`) — not testable against your local PHP backend. Several files are debug/one-off migration scripts (`test_db.php`, `get_schema.php`, `combo_forge_init.php`, `migrate.php`) — not part of user-facing flows.

#### 1.5 Theme App Extension (`extensions/cart-drawer/`) — **storefront-facing, no Shopify admin auth needed**
This is the one part of the app reachable with **zero authentication complexity** — it's public storefront code. `cart_drawer.liquid` (loads `cart_drawer_inline.js`), `coupon_slider.liquid` and `Fbt.liquid` (both inline-script custom elements), `star_rating.liquid`. All hit `*.php.jsx` proxy routes via the Shopify app proxy (`/apps/cart-app/...`). **Fully covered.**

#### 1.6 Data layer
Dual DB confirmed: MySQL (`cart_drawer_ninja`, ~25 tables) for widget settings/analytics/billing, Prisma/SQLite for Session storage + a few models (`CartDrawer`, `Coupon`, `UpsellRule`, `ComboSubscription`, etc.), plus raw-SQL-only tables (`combo_templates`, `combo_analytics`) not in the Prisma schema at all.
